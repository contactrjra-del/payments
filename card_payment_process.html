1. Key Players in Payment Ecosystem

Acquirer (Acquiring Bank)

  • Merchant's bank/payment processor
  • Routes transactions to card networks
  • Settles funds to merchant accounts
  • Assumes chargeback risk

Issuer (Issuing Bank)

  • Cardholder's bank
  • Authorizes/declines transactions
  • Manages credit limits & fraud rules
  • Transfers funds to acquirer

2. Transaction Flow

1. CARDHOLDER → presents card at merchant 2. MERCHANT → sends transaction to ACQUIRER 3. ACQUIRER → routes to CARD NETWORK (Visa/Mastercard/Amex) 4. CARD NETWORK → forwards to ISSUER 5. ISSUER → authenticates & authorizes/declines 6. Response flows back: ISSUER → NETWORK → ACQUIRER → MERCHANT 7. Settlement (T+1 to T+3): Funds move ISSUER → NETWORK → ACQUIRER → MERCHANT

3. Card Authentication Methods

  • PIN (Personal Identification Number): Offline/online PIN verification, common in EMV chip transactions
  • Signature: Traditional method, being phased out for most transactions
  • CVV/CVC: 3-4 digit code on card for CNP transactions
  • 3D Secure (3DS): Visa Secure, Mastercard Identity Check - additional authentication layer
  • Biometric: Fingerprint, face recognition for mobile payments
  • Dynamic CVV: Changing security codes on digital/physical cards

4. EMV Chip Technology

EMV Scenarios & Processing

  • Chip & PIN: Most secure - cardholder verification via PIN
  • Chip & Signature: Chip authentication with signature verification
  • Contactless/NFC: Tap-to-pay under transaction limits
  • Fallback to Magstripe: When chip fails (higher fraud risk)
  • Offline vs Online: Offline uses issuer scripts; online connects to issuer in real-time
  • Dynamic Data: Each transaction generates unique cryptogram preventing cloning

5. Card Present vs Card Not Present

Card Present (CP)

  • Environment: Physical retail, POS terminals
  • Methods: Chip, magstripe, contactless
  • Verification: PIN, signature, biometric
  • Fraud Risk: Lower (physical card present)
  • Interchange: Lower rates
  • Liability: Typically with issuer (EMV shift)

Card Not Present (CNP)

  • Environment: E-commerce, phone orders, recurring
  • Methods: Manual entry, saved cards, tokens
  • Verification: CVV, AVS, 3DS, device fingerprinting
  • Fraud Risk: Higher (no physical verification)
  • Interchange: Higher rates
  • Liability: Typically with merchant

6. Tokenization & Detokenization

Tokenization Process

1. Card PAN (Primary Account Number) entered 2. Payment gateway/processor generates unique token 3. Token stored in merchant system 4. Original PAN stored in secure token vault 5. Token used for subsequent transactions 6. PCI compliance scope reduced significantly

Detokenization Process

1. Merchant submits token for transaction 2. Token service provider validates token 3. Original PAN retrieved from secure vault 4. PAN sent to payment network for processing 5. Token remains in merchant system for future use
Token Types:
  • Payment Tokens: Network tokens (Visa, Mastercard) for recurring payments
  • Gateway Tokens: Processor-specific tokens
  • Mobile Wallet Tokens: Apple Pay, Google Pay device-specific tokens
  • Benefits: Reduced PCI scope, enhanced security, prevents data breaches

7. Mobile Payments

Wallet-Based (NFC)

  • Apple Pay, Google Pay, Samsung Pay
  • Tokenized card credentials
  • Biometric authentication
  • Device-specific tokens
  • Works at contactless terminals

QR Code & App-Based

  • Merchant/customer presented QR
  • Popular in Asia (Alipay, WeChat)
  • Lower infrastructure requirements
  • In-app payment SDKs
  • Direct bank account linking
Mobile Payment Security Features:
  • Tokenization replacing actual card numbers
  • Dynamic transaction cryptograms
  • Device fingerprinting & risk analysis
  • Biometric authentication (Touch ID, Face ID)
  • Secure Element (SE) or Host Card Emulation (HCE)

8. Security Framework

PCI DSS Compliance

  • 12 requirements across 6 goals for securing cardholder data
  • Network security, access controls, encryption
  • Regular testing and monitoring
  • 4 levels based on transaction volume

Fraud Prevention Mechanisms

Real-Time Controls

  • CVV/AVS verification
  • Velocity checks
  • Geolocation matching
  • Device fingerprinting
  • Behavioral biometrics

Advanced Technologies

  • Machine learning fraud models
  • Neural networks for pattern detection
  • Consortium data sharing
  • Risk scoring engines
  • Real-time rule engines

Encryption Standards

  • TLS 1.2+: Secure data transmission
  • AES-256: Data at rest encryption
  • P2PE (Point-to-Point Encryption): End-to-end encryption from terminal
  • Tokenization: Replacing sensitive data with non-sensitive equivalents

9. Additional Important Topics

Authorization vs Settlement

  • Authorization: Real-time approval/decline, holds funds
  • Clearing: Transaction details exchanged between parties
  • Settlement: Actual fund transfer (typically T+1 to T+3 days)
  • Batch Processing: End-of-day transaction submission

Chargebacks & Disputes

  • Cardholder disputes transaction with issuer
  • Reason codes: fraud, processing errors, consumer disputes
  • Merchant has limited time to respond with evidence
  • High chargeback ratios can lead to penalties/termination

Strong Customer Authentication (SCA)

  • EU regulation requiring two-factor authentication
  • Knowledge (password), possession (device), inherence (biometric)
  • Exemptions for low-value, recurring, trusted beneficiaries
  • 3DS 2.0 primary implementation method

Network Tokens vs Gateway Tokens

  • Network Tokens: Issued by card networks, work across acquirers, support auto-updates
  • Gateway Tokens: Processor-specific, limited portability, faster implementation
  • Both reduce PCI scope and enhance security

10. Card Payment Certifications

PCI DSS Certification

  • Level 1: 6M+ transactions/year - Annual onsite audit by QSA
  • Level 2: 1M-6M transactions/year - Annual SAQ + quarterly network scans
  • Level 3: 20K-1M e-commerce transactions/year - Annual SAQ
  • Level 4: Under 20K e-commerce or 1M total - Annual SAQ
  • SAQ Types: A (redirect), A-EP (e-commerce), D (all others)

EMV Certifications

Terminal Certifications

  • EMVCo Type Approval: Chip reading, contactless
  • L1 (Physical): Electrical/mechanical interface
  • L2 (Protocol): Transaction flow, data exchange
  • L3 (Application): Payment brand specific (Visa, MC)
  • Contactless L1/L2: NFC specifications

Card/Device Certifications

  • EMV Card Certification: Chip OS, applet testing
  • Payment Brand Approval: Visa, Mastercard, Amex specs
  • Mobile Wallet Certification: Token provisioning
  • Common Criteria (CC): Security evaluation
  • FIPS 140-2/3: Cryptographic module validation

Payment Brand Certifications

  • Visa: ADVT (Acquirer Device Validation Toolkit), VAS certification
  • Mastercard: M-TIP (Mastercard Terminal Integration Process), MDES
  • American Express: Terminal certification, ExpressPay contactless
  • Discover: D-PAS (Discover Payment Application Specification)
  • Payment Facilitator: Third-party processor registration requirements

Regional & Security Certifications

  • PCI PTS (PIN Transaction Security): PIN pad/POS device approval
  • PCI P2PE: Point-to-point encryption solution validation
  • PA-DSS (deprecated): Replaced by PCI Secure Software Standard
  • 3DS Server Certification: EMVCo 3DS compliance
  • ISO 27001: Information security management
  • SOC 2 Type II: Service organization controls audit

11. Certification Testing Process

EMV Testing Phases

Phase 1: Pre-Certification (Internal Testing) → Functional testing against EMV specs → Test card scenarios (contact/contactless) → Kernel testing, application selection Phase 2: Lab Testing (Accredited Test House) → L1 Physical layer tests → L2 Protocol compliance tests → L3 Application layer tests → Contactless/NFC certification Phase 3: Payment Brand Testing → Brand-specific requirements → Online/offline authorization → Decline scenarios, fallback testing → Certification approval & listing Phase 4: Field Testing & Deployment → Pilot deployment → Real-world transaction monitoring → Performance validation

PCI DSS Testing Requirements

  • Vulnerability Scans: Quarterly ASV (Approved Scanning Vendor) scans
  • Penetration Testing: Annual internal/external pen tests
  • Security Assessments: QSA (Qualified Security Assessor) audits
  • Network Segmentation: Validation of CDE (Cardholder Data Environment) isolation
  • Code Review: Application security testing for custom software
  • Remediation: Fix findings before attestation

Testing Environments & Tools

Test Cards & Simulators

  • Physical EMV test cards (various profiles)
  • Virtual test card numbers
  • Issuer simulators (approval/decline scenarios)
  • Network simulators (Visa, Mastercard)
  • HSM (Hardware Security Module) for crypto

Test Scenarios

  • Happy path transactions
  • Decline scenarios (insufficient funds, expired)
  • Fallback testing (chip to magstripe)
  • Timeout & error handling
  • Reversal & void transactions
  • Currency conversion testing

Certification Timeline

  • POS Terminal EMV: 4-8 months (all levels + brands)
  • Payment Gateway: 3-6 months (PCI + brand integrations)
  • Mobile Wallet: 6-12 months (tokenization + device certification)
  • PCI DSS Compliance: 3-12 months (depending on remediation scope)
  • Recertification: Typically required every 1-3 years or with major updates

12. API Testing in Card Payments

Payment API Types

Authorization APIs

  • Payment authorization/capture
  • Refund/void transactions
  • Pre-authorization (hotels, rentals)
  • Incremental authorization
  • Real-time decline reasons

Tokenization APIs

  • Token creation (PAN to token)
  • Token lifecycle management
  • Detokenization (token to PAN)
  • Network token provisioning
  • Token update notifications

API Testing Methodology

1. Functional Testing → Validate request/response formats (JSON/XML) → Test all API endpoints → Positive & negative test cases → Error code validation 2. Security Testing → TLS/SSL certificate validation → API key/OAuth token authentication → Encryption validation (in-transit/at-rest) → SQL injection, XSS prevention → Rate limiting & throttling 3. Performance Testing → Load testing (concurrent transactions) → Stress testing (peak scenarios) → Response time validation (<2-3 seconds) → Throughput testing (TPS capacity) 4. Integration Testing → End-to-end transaction flow → Multi-system integration (gateway, processor, bank) → Webhook/callback testing → Idempotency testing

Key API Test Scenarios

  • Authorization Success: Valid card, sufficient funds, approved transaction
  • Decline Scenarios: Insufficient funds, invalid CVV, expired card, fraud decline
  • Network Errors: Timeout handling, retry logic, circuit breaker patterns
  • Idempotency: Duplicate transaction prevention with idempotency keys
  • Webhook Delivery: Payment status updates, retry mechanisms
  • 3DS Flow: Challenge/frictionless flow, authentication response handling
  • Tokenization: Create, retrieve, update, delete tokens
  • Reconciliation: Settlement reports, batch processing

API Testing Tools

Manual Testing

  • Postman: API request building & collections
  • Swagger/OpenAPI: API documentation & testing
  • cURL: Command-line API testing
  • Insomnia: REST/GraphQL client

Automated Testing

  • JMeter: Load & performance testing
  • REST Assured: Java API testing framework
  • Pytest: Python testing framework
  • Newman: Postman CLI runner for CI/CD
  • Gatling: Performance testing tool

API Security Testing Checklist

  • Authentication: API key rotation, OAuth 2.0, JWT validation
  • Authorization: Role-based access control (RBAC), scope validation
  • Input Validation: Parameter sanitization, data type validation
  • Rate Limiting: DDoS protection, request throttling
  • Encryption: TLS 1.2+, sensitive data masking in logs
  • PCI Compliance: No PAN in URLs/logs, tokenization enforcement
  • Error Handling: No sensitive data in error messages
  • CORS: Proper cross-origin resource sharing configuration

Sandbox vs Production Testing

Sandbox Environment

  • Test card numbers provided by processor
  • Simulated authorization responses
  • No real money movement
  • Full API feature parity
  • Used for development & UAT

Production Testing

  • Real card transactions (small amounts)
  • Smoke testing post-deployment
  • Monitoring & alerting validation
  • Synthetic transaction monitoring
  • Immediate refund of test transactions

13. Emerging Trends

  • Biometric Cards: Fingerprint sensors embedded in cards
  • Cryptocurrency Integration: Crypto-to-fiat at point of sale
  • BNPL (Buy Now Pay Later): Alternative payment methods at checkout
  • Open Banking: Account-to-account payments, reducing card dependency
  • AI Fraud Detection: Advanced machine learning models
  • Passwordless Authentication: FIDO standards, passkeys
  • Real-Time Payments: ISO 20022 messaging, instant settlement

Note: This overview provides high-level concepts. Actual implementation varies by region, network, and specific payment processor requirements.
